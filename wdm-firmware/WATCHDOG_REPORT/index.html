    Watchdog System Report -- WDM OBC¶

    Platform: RP2040 + TPS1H200AQDGNRQ1 x5
    Subsystems: IbNav . ADCS . Jetson . TT&C . A7Sii

    1. TPS1H200 Hardware Capabilities¶

    Pin Direction Description
    IN (EN_xxx) RP2040 -> chip HIGH = power ON, LOW = power OFF
    FAULT (F1_xxx) chip -> RP2040 Open-drain, LOW = fault active
    DIAG EN hardwired HIGH Diagnostics always active
    CL hardware (resistor) Current limit set per channel
    DELAY hardware (capacitor) Auto-retry delay after fault (~0.2s)

    2. Per-Channel Parameters¶

    Subsystem EN Pin F1 Pin Output Current Limit Notes
    IbNav GPIO 6 GPIO 5 5V 2A GPS/Nav -- PPS watchable
    ADCS GPIO 12 GPIO 11 5V 1A UART bus -- heartbeat watchable
    Jetson GPIO 18 GPIO 17 15V 4A High power -- thermal risk
    TT&C GPIO 23 GPIO 22 5V 3.3A Comms link -- last to disable
    A7Sii GPIO 24 GPIO 26 5V 1.5A Camera -- non-critical

    2.1 Recommended Disable Priority (fault storm)¶

    1. A7Sii -- non-critical payload
    2. Jetson -- high power, disable first if thermal issue
    3. ADCS -- disable only if OBC stability assured
    4. IbNav -- disable only if GPS not needed
    5. TT&C -- last resort, needed for ground communication

    3. DIAG EN Current Sensing (optional -- hardware mod required)¶

    The TPS1H200 outputs a current on the FAULT pin proportional to the load current when DIAG EN is HIGH. To use this as an ADC input:

    1. Free the DIAG EN pin from hardwire -> connect to a GPIO
    2. Add a sense resistor (e.g. 10kOhm) from FAULT to GND
    3. Connect FAULT to an RP2040 ADC pin (GPIO 26-29)
    4. Read: analogRead(pinFault) -> convert to current

    This enables per-channel power consumption monitoring without external current sensors.

    Note: Requires PCB rework on current hardware since DIAG EN is hardwired.

    4. Fault Scenarios¶

    # Scenario Source FAULT pin Recommended Action
    F1 Overcurrent Load draws > CL limit LOW Power cycle, log fault count
    F2 Overtemperature Junction temp > threshold LOW Disable channel, wait cooldown
    F3 Open load No load detected (DIAG EN active) LOW Log warning, check connection
    F4 Undervoltage on VS Input rail drops LOW Disable channel, alert TT&C
    F5 Repeated fault (>3x) Software counter -- Extended disable, escalate
    F6 Recovery failure (>5x) Software counter -- Permanent disable, flag to TT&C
    F7 OBC loop hang RP2040 WDT timeout -- Full RP2040 reset
    F8 PPS signal lost IbNav timeout -- Power cycle IbNav
    F9 Subsystem silent No heartbeat on CLK/SDA -- Power cycle subsystem

    5. Watchdog Levels¶

    Level 1 -- Hardware (TPS1H200)
          Overcurrent / overtemp -> auto shutdown + auto-retry (0.2s delay cap)

Level 2 -- Software fault detection (RP2040)
          checkAllFaults() lit les F1 sur front montant -> log + incremente
          _faultCounters[ch], puis appelle faultHandler(ch)

Level 3 -- Reponse graduee (escalate(), pilotee par _faultCounters[ch])
          cnt < 5   (FAULT_EXT_THRESHOLD)  -> power cycle   (cycle 500ms, POWER_CYCLE_MS)
          cnt >= 5  (FAULT_EXT_THRESHOLD)  -> disable prolonge 30s NON bloquant
                                             (EXTENDED_DISABLE_MS, re-enable auto)
          cnt >= 10 (FAULT_PERM_THRESHOLD) -> disable definitif + alerte TT&C
                                             (markDisabled, irreversible)

Level 4 -- Recuperation
          Canal sain depuis 60s (RECOVERY_RESET_MS) -> resetFaultCounter(ch)
          Re-enable auto a l'expiration du disable prolonge

Level 5 -- OBC self-watchdog (surveillance inter-coeurs)
          Core 0 appelle notifyAlive() a chaque tour de loop() (horodatage).
          Core 1 (loop1) appelle kick() : nourrit le WDT hardware UNIQUEMENT si
          Core 0 a signale depuis < CORE0_ALIVE_TIMEOUT_MS (4s).
          Si Core 0 hang -> kick() appelle fullSystemReset() -> reboot RP2040.
          Filet de securite ultime : si Core 1 hang aussi, le WDT hardware (8s)
          expire seul et reboot la puce.

    Notes d'implementation : - Le disable prolonge est non bloquant : escalate() pose un timer _disableUntil[ch] et checkAllFaults() re-active le canal a expiration. Aucun delay(30000) (qui ferait expirer le WDT 8s du Core 1). - L'escalade se declenche sur la repetition des fronts montants : les retries auto du TPS1H200 (0.2s) font osciller FAULT -> _faultCounters[ch] monte. - Reboot centralise : fullSystemReset() est le point d'entree unique de reboot (coupe tous les canaux -> watchdog_reboot()). kick() l'appelle quand Core 0 est silencieux, au lieu d'inhiber passivement le WDT. - Surveillance inter-coeurs : _core0AliveAt est volatile (visibilite entre Core 0 et Core 1). isCore0Alive() compare a CORE0_ALIVE_TIMEOUT_MS. - Seuils ajustables dans Watchdog.h (FAULT_EXT_THRESHOLD, FAULT_PERM_THRESHOLD, POWER_CYCLE_MS, EXTENDED_DISABLE_MS, RECOVERY_RESET_MS, HEARTBEAT_TIMEOUT_MS, CORE0_ALIVE_TIMEOUT_MS).

    Fault ASCENT APOGEE DESCENT LANDING
    IbNav fault Power cycle x3 -> alert TT&C Power cycle x3 -> alert Power cycle immediat Log only
    ADCS fault Power cycle -> escalade Power cycle -> escalade Power cycle -> disable N/A (OFF)
    Jetson fault Power cycle x3 -> disable Power cycle x3 -> disable N/A (OFF) N/A (OFF)
    TT&C fault Power cycle x3 -> reset OBC Power cycle x3 Power cycle continu Power cycle continu
    A7Sii fault Power cycle -> disable Power cycle -> disable N/A (OFF) N/A (OFF)

    Etat cible (TODO) : escalate() est actuellement state-agnostic -- il applique la meme reponse graduee (cycle / disable prolonge / disable definitif) quel que soit l'etat de vol. Rendre la reponse dependante de flightState (priorites de la table ci-dessus) reste a implementer.

    8. Power-On Sequence¶

    8.1 Requirement¶

    All modules must not power up simultaneously to avoid inrush current spikes that could trip fuses or brown out the supply.

    Each module is powered individually, followed by a stabilization delay and a communication test to confirm the module responds before moving to the next. On failure, the module is marked disabled and the sequence continues without it.

    8.2 Power-Up Order¶

    Step Module Stabilization Communication Test Test Timeout
    1 TT&C 500 ms Send UART ping to RFD900x, wait ACK 2 s
    2 IbNav 1000 ms Read NMEA bytes on I2C0 (addr IBNAV_I2C_ADDR) 3 s
    3 ADCS 500 ms Wait for any byte on UART1 (raw text stream) 3 s
    4 Jetson 2000 ms Ping I2C address JETSON_I2C_ADDR, wait Wire ACK 5 s
    5 A7Sii 500 ms Check F1 HIGH (no TPS1H200 fault) 500 ms

    TT&C powered first -- required to log boot results to ground.
    Jetson 2 s delay -- Linux boot time on embedded module.

    8.3 Failure Behavior¶

    Test passed  -> log "[BOOT] <module> OK" -> continue sequence
Test failed  -> log "[BOOT] <module> FAIL -- skipped"
             -> module marked DISABLED in watchdog _disabledMask
             -> watchdog configured to skip this module
             -> TT&C alerts ground

    A module that fails the boot test can be retried via TT&C ground command.

    9. Telemetry -- Data Acquisition & Packet¶

    9.1 Overview¶

    The RP2040 OBC collects data from all subsystems, builds a structured packet, and transmits it via RFD900x (TT&C) to ground.

    [IbNav GPS]  --I2C0 (GPIO 0/1)----->
[ADCS]       --UART1 (GPIO 8/9)---->  [RP2040 OBC]  --> Packet Builder --> [RFD900x TT&C] --> Ground
  +- IMU                                                  (UART0, GPIO 28/29)
  +- Baro
  +- 8x Temp
[Jetson]     --I2C1 (Wire1, 14/15)>

    Allocation des peripheriques RP2040 : I2C0 = IbNav, I2C1 = Jetson, UART0 = TT&C (RFD900x), UART1 = ADCS. Les 2 I2C et 2 UART sont distincts.

    9.2 Data Sources¶

    REQ-DAQ-01 -- IbNav GPS (I2C0, GPIO 0/1)¶

    Field Format Rate
    Latitude float, decimal degrees 1 Hz
    Longitude float, decimal degrees 1 Hz
    Altitude (MSL) float, metres 1 Hz
    Ground speed float, m/s 1 Hz
    Heading float, degrees 1 Hz
    Fix quality uint8 (0=none, 1=GPS, 2=DGPS) 1 Hz
    Satellites uint8 1 Hz
    PPS health bool (pulse present < 1100ms) continuous

    Protocol: NMEA 0183 (GGA, RMC sentences) lu via le bus I2C/DDC du module. Adresse I2C : IBNAV_I2C_ADDR (TODO a confirmer). 1PPS materiel sur GPIO 4.

    REQ-DAQ-02 -- ADCS (UART1, GPIO 8 TX / 9 RX)¶

    The ADCS has its own processor that aggregates IMU, barometer, and 8 temperature probes, then streams a structured UART packet to the RP2040.

    Field Format Rate
    Accelerometer X/Y/Z float, m/s^2 10 Hz
    Gyroscope X/Y/Z float, deg/s 10 Hz
    Roll / Pitch / Yaw float, degrees 10 Hz
    Absolute pressure float, hPa 1 Hz
    Barometric altitude float, metres 1 Hz
    Temp[0..7] float x8, degC 0.5 Hz

    TEMPORARY FORMAT -- to be replaced per protocol agreed with ADCS team.
    Baud rate: 115200. Liaison UART en texte brut (raw), pas de paquet binaire.
    Pas de sync bytes : sur une liaison UART point a point, les octets sont lus directement via ADCS::forwardRaw(). Le parsing en champs viendra avec collectADCS() une fois le format texte confirme.
    GPIO 7 (CLK_ADCS) used as sync/interrupt signal from ADCS.
    GPIO 10 (RUN_ADCS): start/stop command for ADCS processor.

    REQ-DAQ-05 -- Jetson (I2C1 -- Wire1, GPIO 14/15)¶

    Field Format Rate
    Operational status uint8 (enum) 1 Hz
    CPU load uint8, % 1 Hz
    GPU/CPU temperature float, degC 1 Hz
    Science data (summary) bytes[N] on event

    Dedicated Wire1 bus (I2C1) -- isolated from ADCS bus to avoid conflicts.
    GPIO 13 (CLK_Jetson) used as sync/interrupt/heartbeat from Jetson.
    Heartbeat absence > 5s -> watchdog reacts.
    I2C address to confirm with Jetson firmware.

    REQ-DAQ-06 -- OBC Health (RP2040 internal)¶

    Field Format Rate
    Flight state uint8 (enum) 1 Hz
    Fault flags (5 bits) uint8 bitmask 1 Hz
    Uptime uint32, seconds 1 Hz
    Battery voltage float, V (ADC) 0.5 Hz
    RP2040 temperature float, degC (internal ADC) 0.5 Hz

    9.4 RFD900x Transmission¶

    Parameter Value
    Interface UART0 -> TX GPIO 28 / RX GPIO 29
    Baud rate 57600 bps (TTC_BAUD)
    TX frequency 1 Hz nominal, 0.2 Hz in LANDING
    Mode Transparent serial (no MAVLink required)
    Packet size 124 bytes -> ~17ms at 57600 bps
    Redundancy Packet repeated 2x if no ground ACK within 2s

    9.5 Repartition dual-core (implementee)¶

    Core 0 (loop()) : collecte raw + telemetrie + terminal. Chaque subsystem gere son flux et recopie ses octets vers Serial sans parsing. Core 1 (loop1()) : surveillance Watchdog dediee, independante de Core 0.

    void loop() {                 // ---- Core 0 ----
  watchdog.notifyAlive();     // signale la vitalite a Core 1 (surveillance croisee)
  terminal.poll();            // lecture non bloquante des commandes (USB/TT&C)

  // 1) Recopie raw de chaque subsystem vers Serial (debug)
  ibNav.forwardRaw();         // trames NMEA (I2C0, sondage 1 Hz)
  adcs.forwardRaw();          // flux ADCS  (UART1)
  jetson.forwardRaw();        // sondage I2C1 a 1 Hz

  // 2) Assemblage + transmission telemetrie au sol a 1 Hz
  static uint32_t lastTx = 0;
  if (millis() - lastTx >= 1000) {
    lastTx = millis();
    TelemetryPacket pkt = {};        // zero-init : champs non collectes = 0
    telemetry.collectOBCHealth(pkt); // etat de vol, uptime, fautes, batterie
    telemetry.buildPacket(pkt);      // sync 0xAA55 + version + CRC-16
    telemetry.transmit(pkt);         // 124 octets vers RFD900x (TT&C)
  }
}

void loop1() {                // ---- Core 1 ----
  watchdog.kick();            // nourrit le WDT SI Core 0 vivant, sinon fullSystemReset
  watchdog.checkAllFaults();  // poll des 5 pins FAULT + escalade

  // Surveillance CLK : ADCS(1)/Jetson(2)/A7Sii(4) generent un signal d'activite.
  // IbNav(0)/TTC(3) renvoient true (pas de CLK en entree).
  for (uint8_t i = 0; i < 5; i++)
    if (!watchdog.monitorHeartbeat(i)) watchdog.faultHandler(i);

  delay(1000);                // Core 1 dedie : delay acceptable a 1 Hz
}

    Securite inter-coeurs : si loop() (Core 0) se bloque, notifyAlive() n'est plus appele ; apres CORE0_ALIVE_TIMEOUT_MS (4s), kick() cesse de nourrir le WDT et declenche fullSystemReset(). Detection d'un blocage Core 0 en ~4-8s. Si les deux coeurs gelent, le WDT hardware (8s) reboot seul.

    Source Methode Mecanisme Liaison
    IbNav (GPS) IbNav::forwardRaw() requestFrom(IBNAV_ADDR, 32) @ 1 Hz (timer interne), puis recopie I2C0
    ADCS ADCS::forwardRaw() while (available()) Serial.write(read()) UART1, texte brut
    Jetson Jetson::forwardRaw() requestFrom(0x42, 8) @ 1 Hz (timer interne), puis recopie I2C1
    Telemetrie collectOBCHealth -> buildPacket -> transmit timer 1 Hz, paquet 124 B UART TT&C (RFD900x)

    Flux en texte brut : pas de sync bytes, pas d'accumulation, pas de parsing binaire. Les octets recus (UART ADCS, ou I2C IbNav/Jetson) sont recopies tels quels. L'ancienne structure binaire ADCSPacket et les sync bytes 0xAD 0xC5 ont ete retires.

    Non bloquant : IbNav::forwardRaw() et Jetson::forwardRaw() embarquent chacun leur propre timer millis() 1 Hz (sondage I2C), donc loop() ne bloque jamais. ADCS lit son UART de maniere non bloquante (available()).

    Telemetrie implementee : collectOBCHealth() remplit flightState + uptime + faultFlags ; buildPacket() ajoute sync/version/CRC-16 ; transmit() envoie les 124 octets via TTC::transmit() sur l'UART RFD900x.

    Prochaine etape : implementer collectGPS / collectADCS / collectJetson (parsing des flux texte vers les champs de TelemetryPacket) et la lecture battVoltage via une broche ADC dediee.

    10. Flight State Machine¶

    10.1 States & Transitions¶

    [BOOT] --> [ASCENT] --> [APOGEE] --> [DESCENT] --> [LANDING]
    State Trigger Detection
    BOOT Power on Startup init complete
    ASCENT After boot Vertical speed > +1 m/s sustained
    APOGEE Peak altitude Vertical speed crosses 0, altitude max
    DESCENT Balloon burst Vertical speed < -2 m/s sustained
    LANDING Ground contact Altitude < threshold + speed ~ 0 for >30s

    Detection source: IbNav GPS altitude + computed vertical speed (Deltaaltitude / Deltatime)

    10.2 Subsystem Power Profile per State¶

    Subsystem BOOT ASCENT APOGEE DESCENT LANDING
    IbNav ON ON ON ON ON
    ADCS OFF ON ON ON OFF
    Jetson OFF ON ON OFF OFF
    TT&C ON ON ON ON ON
    A7Sii OFF ON ON OFF OFF

    TT&C always active -- critical ground link in all phases.
    IbNav always active -- needed for localization and recovery.

    10.3 Phase Details¶

    BOOT¶

    • Estimated duration: < 30s
    • Objective: GPIO init, fault check, TT&C handshake with ground
    • Watchdog: active but tolerant (long delays, no power cycles during init)

    ASCENT¶

    • Estimated duration: 90-120 min (0 -> ~30 km)
    • All subsystems active
    • A7Sii recording
    • Jetson processing science data
    • TT&C telemetry rate: nominal (1 Hz)
    • Watchdog: fully active, all faults monitored
    • IbNav PPS monitored: GPS loss -> immediate alert

    APOGEE¶

    • Estimated duration: a few minutes around altitude peak
    • Detection: vertical speed ~ 0 +/- 0.5 m/s over 10s sliding window
    • A7Sii: highest priority (maximum field of view)
    • TT&C: burst telemetry -- transmit compressed science data
    • Watchdog: normal

    DESCENT¶

    • Estimated duration: 30-60 min (burst -> ~5 km/min)
    • Jetson OFF -> power and thermal savings
    • A7Sii OFF
    • ADCS active for parachute stabilization
    • TT&C: recovery beacon active (GPS position repeated)
    • Watchdog: IbNav critical, fault -> immediate power cycle

    LANDING¶

    • Detection: altitude < 500m + speed < 1 m/s for 30 consecutive seconds
    • Jetson OFF, ADCS OFF, A7Sii OFF
    • IbNav: broadcast position every 10s via TT&C
    • Power: maximum savings mode
    • Watchdog: relaxed (no aggressive power cycles on ground)
    • RP2040 WDT: always active to prevent freeze during recovery wait

    11. State Detection Strategy¶

    11.1 Problem¶

    The TPS1H200 FAULT pins are purely binary: HIGH (nominal) or LOW (fault). They give no direct information about altitude or speed. The strategy is to cross-correlate multiple binary sources and apply temporal analysis to infer flight state.

    11.2 Binary Signals Available on OBC¶

    Signal GPIO Type What it can indicate
    PPS_IbNav 4 Pulse 1 Hz GPS lock active -> valid position
    F1_IbNav 5 Fault LOW IbNav power problem
    F1_ADCS 11 Fault LOW ADCS power problem
    F1_Jetson 17 Fault LOW Jetson overcurrent (CPU load)
    F1_TTC 22 Fault LOW TT&C power problem
    F1_A7Sii 26 Fault LOW Camera power problem

    11.3 Detection Techniques¶

    A -- PPS Temporal Analysis (GPIO 4)¶

    The GPS PPS signal is a precise 1 Hz pulse when fix is valid.

    Measure: interval between two rising edges of PPS

PPS present + stable interval ~1000ms -> GPS lock -> altitude available via IbNav I2C
PPS absent                             -> GPS lost -> fall back to time-based

    Implementation: pulseIn(PIN_PPS_IBNAV, HIGH) or rising-edge interrupt + timestamp.

    B -- Fault Count on Sliding Window¶

    A single binary fault means little. Its frequency over a time window is informative:

    10s window -- Jetson fault count:
  0 faults   -> subsystem stable, normal load
  1-2 faults -> transient overload (intensive compute -> apogee?)
  >3 faults  -> real problem -> watchdog escalates

    C -- Cross-Channel Correlation¶

    Flight state generates consistent load patterns across multiple subsystems simultaneously:

    Observed pattern Probable state
    All nominal, PPS stable Normal ASCENT
    Jetson transient fault + A7Sii nominal Apogee data processing
    F1_Jetson LOW permanent + PPS ok Jetson overconsumption -> descent (cold, low battery)
    F1_ADCS LOW after shock LANDING (parachute impact)
    All nominal + PPS lost Altitude > 40 km (GPS signal loss) -- near apogee

    D -- Fault Duration Analysis (time-in-fault)¶

    Brief fault  (< 200ms)  -> transient, TPS1H200 auto-retry sufficient
Medium fault (200ms-2s) -> overload, software power cycle
Long fault   (> 5s)     -> failure -> escalate + change power profile

    Measure with millis() at F1 falling edge and rising edge.

    E -- Hybrid State Machine: binary + time¶

    Without parsed GPS, state can be inferred via mission timer + subsystem health:

    t=0          -> BOOT
t > 60s + all subsystems nominal        -> ASCENT (assumed)
t > 5400s (90 min) OR repeated Jetson faults -> APOGEE (assumed)
Jetson permanent fault + brief ADCS fault    -> DESCENT (burst detected)
Brief F1_ADCS LOW + all silent > 30s         -> LANDING

    This mode is the fallback if IbNav I2C is not yet parsed.

    11.4 Recommended Detection Architecture¶

    Level 1 (primary)   : Parse IbNav I2C -> altitude + vspeed GPS
                      Accurate ASCENT/APOGEE/DESCENT/LANDING detection

Level 2 (secondary) : PPS timing -> confirms GPS lock
                      Consistent with Level 1

Level 3 (fallback)  : Mission timer + binary fault patterns
                      Used if IbNav I2C silent or GPS lost

2-of-3 vote         : If Level 1 and Level 3 diverge -> keep current state,
                      log divergence via TT&C, do not change state

    RP2040 internal temp Internal ADC Altitude correlation (cold = high altitude)
    Battery voltage ADC GPIO 26-29 Power health, coherent with flight phase
