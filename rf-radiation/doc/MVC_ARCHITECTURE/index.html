    RF Link Budget Simulator v2 - MVC Architecture¶

    Overview¶

    This project implements a Model-View-Controller (MVC) architecture to reduce code coupling and organize features independently.

    Architecture Layers¶

    server2/
├── models/                 # Data management layer
│   ├── __init__.py
│   ├── antenna_model.py           # Antenna configurations & patterns
│   ├── telemetry_model.py         # XLSX telemetry data
│   ├── rf_budget_model.py         # RF parameters storage
│   └── footprint_model.py         # Footprint grid calculations
│
├── controllers/            # Business logic & request handling
│   ├── __init__.py
│   ├── footprint_controller.py    # Footprint calculation orchestration
│   ├── antenna_controller.py      # Antenna operations
│   ├── rf_budget_controller.py    # RF parameter management
│   ├── telemetry_controller.py    # Telemetry file operations
│   └── link_budget_controller.py  # RF link budget calculations
│
├── services/              # Shared utilities & business logic
│   ├── __init__.py
│   └── shared_service.py          # CoordinateService, AntennaService, RFService
│
├── static/                # Frontend assets
│   ├── js/
│   └── css/
│
├── app.py                 # Flask application & routes
├── server_2.py           # Legacy monolithic code (kept for reference)
└── index2.html           # Frontend HTML

    Models Layer¶

    Purpose: Data management and persistence

    AntennaModel¶

    Manages antenna patterns, configurations, and gain calculations.

    from server2.models import AntennaModel

antenna = AntennaModel()
configs = antenna.get_configs()           # Get all antenna types
gain_db = antenna.get_gain("yagi", 30)   # Get gain at elevation angle

    Antenna Types: - dipole_half_wave: Basic half-wave dipole (omni) - yagi: Directional yagi antenna - patch: Microstrip patch antenna - helix: Circular polarized helix - omni: Omnidirectional (reference)

    TelemetryModel¶

    Handles telemetry data loading and access from XLSX files.

    telemetry = TelemetryModel()
telemetry.load_from_xlsx("tm_test.xlsx")    # Load data
point = telemetry.get_point(idx)            # Get specific point
all_data = telemetry.get_all()             # Get all points

    RFBudgetModel¶

    Stores and manages RF budget parameters.

    rf_budget = RFBudgetModel()
power_dbm = rf_budget.get("tx_power_dbm")           # Get parameter
rf_budget.set("tx_power_dbm", 10)                   # Set parameter
rf_budget.update({"tx_power_dbm": 10, ...})        # Update multiple

    Key Parameters: - TX Power: tx_power_dbm (0-30 dBm) - RX Sensitivity: rx_sensitivity_dbm (-110 to -100 dBm) - Frequencies: frequency_mhz, uplink_mhz, downlink_mhz - Losses: cable_loss_db, atmospheric_loss_db

    FootprintModel¶

    Calculates power grids with antenna orientation consideration.

    footprint = FootprintModel()
grid_result = footprint.calculate_grid(
    lat=48.5, lon=-81.3, alt=10000,
    pitch=10, roll=5, yaw=45,
    tx_antenna="yagi",
    frequency_mhz=145.0,
    rf_budget={...}
)

    Controllers Layer¶

    Purpose: Business logic and request orchestration

    FootprintController¶

    Orchestrates footprint calculation by coordinating model and service calls.

    from server2.controllers import FootprintController
from server2.models import FootprintModel, AntennaModel

footprint_model = FootprintModel()
antenna_model = AntennaModel()
controller = FootprintController(footprint_model, antenna_model)

result, status = controller.calculate(
    lat=48.5, lon=-81.3, alt=10000,
    pitch=10, roll=5, yaw=45,
    frequency_mhz=145.0,
    tx="yagi",
    grid_n=25,
    rf_budget={...}
)

    LinkBudgetController¶

    Calculates RF link budget for all telemetry points.

    link_budget, status = controller.calculate_link_budget(
    tx="yagi",
    rx="dipole_half_wave",
    gs_lat=48.567, gs_lon=-81.365, gs_alt=285
)

    Services Layer¶

    Purpose: Shared utilities and cross-cutting concerns

    CoordinateService¶

    Coordinate system transformations.

    from server2.services import CoordinateService

# Geodetic to local
east, north, up = CoordinateService.geodetic_to_enu(
    lat_deg=48.5, lon_deg=-81.3, alt_m=10000,
    ref_lat_deg=48.567, ref_lon_deg=-81.365, ref_alt_m=285
)

# Distance, elevation, azimuth
dist, elev, az = CoordinateService.distance_elevation_azimuth(
    lat_deg=48.5, lon_deg=-81.3, alt_m=10000,
    gs_lat_deg=48.567, gs_lon_deg=-81.365, gs_alt_m=285
)

    RFService¶

    RF calculations (FSPL, RSSI, etc).

    from server2.services import RFService

fspl = RFService.free_space_path_loss(freq_mhz=145.0, distance_m=100000)
rssi = RFService.calculate_rssi(
    eirp_dbm=10,
    path_loss_db=120,
    atm_loss_db=2,
    antenna_gain_db=10,
    cable_loss_db=2
)

    Starting the Server¶

python3 main.py

# Or directly
cd server2
python3 -m app

# With environment variable
export FLASK_ENV=production
python3 main.py

    Server runs on http://0.0.0.0:5050

    Benefits of MVC Architecture¶

    1. Separation of Concerns: Each layer has a single responsibility
    2. Reduced Coupling: Models don't depend on Controllers, Controllers don't depend on Views
    3. Feature Isolation: New features can be added without affecting existing code
    4. Testability: Each component can be tested independently
    5. Maintainability: Changes are localized to specific components
    6. Scalability: Easy to add new antenna types, RF parameters, etc.

    Migration from Monolithic Code¶

    The original server_2.py contained: - ~973 lines of mixed code - Antenna patterns, telemetry loading, RF calculations, and routing all together - High coupling between features

    The new MVC structure: - Models: ~400 lines (antenna_model, telemetry_model, rf_budget_model, footprint_model) - Controllers: ~350 lines (orchestration logic) - Services: ~150 lines (shared utilities) - Routes: ~200 lines (Flask app.py) - Total: ~1100 lines (better organized, not more code)
