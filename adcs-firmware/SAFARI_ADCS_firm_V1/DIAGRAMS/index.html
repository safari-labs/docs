    SAFARI ADCS — PlantUML Diagrams & Explanations¶

    This document provides visual representations of the SAFARI ADCS firmware architecture using PlantUML.

    1. System Architecture Overview¶

    Shows all major modules and their interconnections within the RP2040.

    @startuml SAFARI_ADCS_Architecture
!define RECTANGLE_COLOR #E1F5FE
!define PROCESS_COLOR #B3E5FC
!define EXTERNAL_COLOR #FFF9C4

skinparam rectangle {
  BackgroundColor RECTANGLE_COLOR
  BorderColor #01579B
  FontColor #01579B
}

package "RP2040 Microcontroller" {
  rectangle "Main Orchestrator\n(main.cpp)" as main {
  }

  package "Communication Interfaces" {
    rectangle "I2C Bus\n(GPIO 4, 5 @ 400kHz)" as i2c
    rectangle "UART0 Serial\n(GPIO 0, 1 @ 115200)" as serial
  }

  package "Hardware Modules" {
    rectangle "GNSS Module\n(gnss.h/cpp)" as gnss
    rectangle "Sensor Manager\n(sensors.h/cpp)" as sensorMgr
    rectangle "Watchdog Module\n(watchdog.h/cpp)" as watchdog
    rectangle "Status Display\n(printStatus.h/cpp)" as display
  }

  package "Concrete Sensors" {
    rectangle "Pt1000\nThermometer" as pt1000
    rectangle "IMU\n(6-axis)" as imu
    rectangle "Altimeter\n(Barometric)" as altimeter
    rectangle "Sensor Slot 4\n(Reserved)" as slot4
  }

  package "Configuration" {
    rectangle "config.h\n(Pins, Addresses, Constants)" as config
  }
}

package "External Hardware" {
  rectangle "M9N u-blox\nGNSS Receiver\n(I2C 0x42)" as m9n
  rectangle "MPU-6050 or\nICM-42688P\n(I2C 0x68)" as imuHW
  rectangle "BMP280 or\nMS5611\n(I2C 0x77)" as altimeterHW
  rectangle "PT1000 RTD\nwith MAX31865\n(ADC/SPI)" as pt1000HW
  rectangle "External Watchdog\nController" as watchdogHW
}

main --> gnss
main --> sensorMgr
main --> watchdog
main --> display

gnss --> i2c

i2c --> altimeterHW

@enduml

    Explanation:¶

    • Main Orchestrator: Central loop (main.cpp) controls all subsystems
    • Communication Buses: I2C for all sensors (400 kHz), UART for debug/status
    • Modules: GNSS, Sensor Manager, Watchdog, and Status Display operate independently but coordinated
    • Hardware Devices: Each module interfaces with physical devices via I2C/ADC
    • Configuration: Central config.h provides all tunable parameters

    2. Sensor Framework Class Hierarchy¶

    Demonstrates the polymorphic sensor architecture.

    @startuml Sensor_ClassDiagram
skinparam classBackgroundColor #E8F5E9
skinparam classArrowColor #2E7D32
skinparam classBorderColor #2E7D32

abstract class Sensor {
  {abstract} +begin(): bool
  {abstract} +update(): void
  {abstract} +getData(): SensorData
}

class SensorData {
  +value: float
  +valid: bool
  +timestamp: uint32_t
}

struct Pt1000Data {
  +tempC: float
  +resistanceOhm: float
}

struct IMUData {
  +accelX: float
  +accelY: float
  +accelZ: float
  +gyroX: float
  +gyroY: float
  +gyroZ: float
  +tempC: float
}

struct AltimeterData {
  +pressureHPa: float
  +altitudeM: float
  +tempC: float
}

class Pt1000Sensor {
  -data: Pt1000Data
  +begin(): bool
  +update(): void
  +getData(): SensorData
  +getPt1000(): const Pt1000Data&
}

class IMUSensor {
  -data: IMUData
  +begin(): bool
  +update(): void
  +getData(): SensorData
  +getIMU(): const IMUData&
}

class AltimeterSensor {
  -data: AltimeterData
  -seaLevelPressure: float
  +begin(): bool
  +update(): void
  +getData(): SensorData
  +getAltimeter(): const AltimeterData&
}

class SensorManager {
  -sensors[4]: Sensor*
  -count: uint8_t
  +addSensor(sensor: Sensor*): bool
  +updateAll(): void
  +getSensorData(idx: uint8_t): SensorData
  +getSensor(idx: uint8_t): Sensor*
  +getCount(): uint8_t
}

Sensor <|-- Pt1000Sensor
Sensor <|-- IMUSensor
Sensor <|-- AltimeterSensor

Pt1000Sensor --> Pt1000Data
IMUSensor --> IMUData
AltimeterSensor --> AltimeterData

Sensor --> SensorData
SensorManager --> "1..4" Sensor
SensorManager --> SensorData

@enduml

    Explanation:¶

    • Sensor (base class): Defines interface that all sensors must implement
    • Concrete Sensors: Pt1000Sensor, IMUSensor, AltimeterSensor each inherit from Sensor
    • Type-specific Data: Each sensor has its own data struct (Pt1000Data, IMUData, AltimeterData)
    • Generic Adapter: Each concrete class implements getData() returning SensorData for uniform display
    • SensorManager: Manages up to 4 sensor instances, can call updateAll() to refresh all simultaneously

    3. GNSS Module Data Flow¶

    Details the GNSS receiver communication and NMEA parsing pipeline.

    @startuml GNSS_DataFlow
skinparam rectangle {
  BackgroundColor #FFF3E0
  BorderColor #E65100
}

rectangle "GNSS Module (gnss.cpp)" {

  rectangle "I2C Protocol Layer" {
    rectangle "Read Register 0xFD\n(Byte Count)" as reg_fd
    rectangle "Read Register 0xFE\n(Byte Count Hi)" as reg_fe
    rectangle "Read I2C Data\n(up to 255 bytes)" as read_data
  }

  rectangle "NMEA Processing" {
    rectangle "Feed bytes to\nState Machine" as state_machine
    rectangle "Identify Sentence\nType" as identify
    rectangle "Parse NMEA\nSentence" as parse
  }

  rectangle "Sentence Parsers" {
    rectangle "parseGGA()\nPosition, Fix, Sats" as parser_gga
    rectangle "parseRMC()\nDate, Speed, Course" as parser_rmc
    rectangle "parseGSA()\nFix Type (2D/3D)" as parser_gsa
  }

  rectangle "Data Storage" {
    rectangle "Update GPSData\nStruct" as update_struct
  }
}

rectangle "M9N Receiver\n(External Hardware)" {
  rectangle "NMEA Sentence\nBuffer" as buffer
}

buffer --> reg_fd
buffer --> reg_fe
reg_fd --> read_data
reg_fe --> read_data

read_data --> state_machine
state_machine --> identify

identify --> parser_gga
identify --> parser_rmc
identify --> parser_gsa

parser_gga --> parse
parser_rmc --> parse
parser_gsa --> parse

parse --> update_struct

update_struct --> "gnss.getData()"

note right of reg_fd
  Polls byte count
  from receiver
end note

note right of state_machine
  0xFF padding bytes
  skipped automatically
end note

note right of parser_gga
  Extracts: Lat, Lon, Alt
  Fix quality, Satellite count
end note

@enduml

    Explanation:¶

    • I2C Protocol: Queries M9N byte count registers, reads up to 255 bytes per cycle
    • NMEA State Machine: Processes incoming bytes, identifies sentence boundaries
    • Sentence Identification: GGA (position/fix), RMC (date/speed), GSA (fix type)
    • Parsers: Each parser extracts relevant fields using field delimiters and coordinates conversion
    • Data Storage: Parsed values update the shared GPSData struct accessible to main loop

    4. Main Loop Execution Flow¶

    Sequence diagram showing one iteration of the main control loop.

    @startuml Main_Loop_Sequence
participant "main.cpp\nLoop" as loop
participant "GNSS Module" as gnss
participant "Sensor Manager" as sensors
participant "Display Module" as display
participant "Watchdog Module" as watchdog

loop Every ~50ms

  loop Continuous Update
    loop --> gnss: update()
    gnss --> loop: (~10ms)

    loop --> sensors: updateAll()
    sensors --> loop: (< 5ms)
  end

  alt Every 1 second (1Hz)
    loop --> display: printStatus()
    display --> loop: (~20ms)
    note right of display
      Builds 4-column table,
      sends to Serial
    end note
  end

  alt Every 5 seconds
    loop --> watchdog: sendHeartbeat()
    watchdog --> loop: (<1ms)
    note right of watchdog
      Sends "alive" signal
      to external watchdog
    end note
  end

  loop --> loop: delay(10ms)\n(yield to OS)

end

note over loop, watchdog
  Nominal cycle: ~50ms
  Margin: ~15ms for anomalies
  Total time budget: 50ms
end note

@enduml

    Explanation:¶

    • Continuous Update: GNSS and Sensor updates run every cycle (~10 ms for GNSS, <5 ms for sensors)
    • 1 Hz Display: Status printed to serial every second without blocking
    • 5 Hz Watchdog: Heartbeat sent every 5 seconds to external system
    • Timing Budget: 50 ms cycle with 15 ms margin maintains real-time responsiveness

    5. I2C Bus Topology¶

    Physical layout of all I2C devices on the RP2040.

    @startuml I2C_Topology
skinparam rectangle {
  BackgroundColor #E0E0E0
  BorderColor #424242
}

rectangle "RP2040 I2C Bus" {
  rectangle "GPIO 4 (SDA)\nGPIO 5 (SCL)\n400 kHz" as bus_control
}

rectangle "Connected Devices" {
  rectangle "M9N GNSS\nI2C Address: 0x42\n(UART protocol via I2C)" as m9n_dev
  rectangle "IMU\nI2C Address: 0x68\n(MPU-6050, ICM-42688P, LSM6DS3)" as imu_dev
  rectangle "Altimeter\nI2C Address: 0x77\n(BMP280, MS5611, LPS22HB)" as alt_dev
}

rectangle "Other Interfaces" {
  rectangle "PT1000 RTD\nADC or SPI\n(MAX31865)" as pt1000_dev
}

bus_control --> m9n_dev
bus_control --> imu_dev
bus_control --> alt_dev

note right of m9n_dev
  Pull-up resistors on
  SDA/SCL required
end note

note right of imu_dev
  Address configurable
  via AD0 pin
end note

note right of alt_dev
  Address configurable
  via SDO pin
end note

note right of pt1000_dev
  Separate from I2C
  Uses ADC or dedicated SPI
end note

@enduml

    Explanation:¶

    • I2C Master: RP2040 acts as master on GPIO 4 (SDA) and GPIO 5 (SCL)
    • Bus Speed: 400 kHz suitable for all connected devices
    • Device Addresses: M9N at 0x42, IMU at 0x68, Altimeter at 0x77
    • PT1000: Uses separate ADC or SPI interface (not on I2C bus)
    • Extensibility: Address lines can be configured to support multiple instances

    7. Sensor Update Sequence¶

    Detailed timeline of how each sensor is queried and updated.

    @startuml Sensor_Update_Sequence
participant "Main Loop" as main
participant "SensorMgr" as mgr
participant "Pt1000Sensor" as pt1000
participant "IMUSensor" as imu
participant "AltimeterSensor" as alt

main --> mgr: updateAll()

mgr --> pt1000: update()
activate pt1000
pt1000 --> pt1000: Read ADC/SPI
pt1000 --> pt1000: Convert to Temperature
pt1000 --> pt1000: Store in Pt1000Data
deactivate pt1000

mgr --> imu: update()
activate imu
imu --> imu: Read I2C (0x68)
imu --> imu: Parse accel/gyro/temp
imu --> imu: Store in IMUData
deactivate imu

mgr --> alt: update()
activate alt
alt --> alt: Read I2C (0x77)
alt --> alt: Parse pressure
alt --> alt: Convert to altitude
alt --> alt: Store in AltimeterData
deactivate alt

mgr --> main: return

note right of mgr
  Total time: < 5ms
  (stub implementations)
end note

@enduml

    Explanation:¶

    • Sequential Update: SensorManager calls update() on each sensor in order
    • Pt1000: Reads ADC/SPI and converts resistance to temperature
    • IMU: Queries I2C @ 0x68 for 6-axis accel/gyro + die temperature
    • Altimeter: Reads pressure from I2C @ 0x77 and calculates barometric altitude
    • Timing: All sensors update in <5 ms per cycle

    8. Watchdog Communication Protocol¶

    Interaction between RP2040 and external watchdog controller.

    @startuml Watchdog_Protocol
participant "Main Loop\n(RP2040)" as rp2040
participant "External Watchdog\nController" as watchdog

rp2040 --> watchdog: sendHeartbeat()\nevery 5 seconds

activate watchdog
watchdog --> watchdog: Update last_heartbeat timestamp
watchdog --> watchdog: Check timeout threshold
watchdog --> watchdog: Update WatchdogStatus
deactivate watchdog

alt Normal Operation
  watchdog --> rp2040: OK (healthy)
else Timeout Detected
  watchdog --> rp2040: ALERT (RP2040 hung)
  watchdog --> watchdog: Trigger watchdog action\n(reset, power cycle, etc.)
end

rp2040 --> watchdog: sendAlert(code)\nif emergency detected

activate watchdog
watchdog --> watchdog: Log alert code
watchdog --> watchdog: Take emergency action
deactivate watchdog

rp2040 --> watchdog: isWatchdogAlive()\nquery status

activate watchdog
watchdog --> rp2040: return WatchdogStatus
deactivate watchdog

note right of rp2040
  Heartbeat interval: 5 seconds
  Watchdog timeout: ~10-15 seconds (TBD)
  Protocol: TBD (UART, I2C, CAN)
end note

@enduml

    Explanation:¶

    • Heartbeat (5 sec): RP2040 sends periodic "alive" signal to watchdog
    • Timeout Detection: Watchdog tracks last heartbeat; triggers if not received within threshold
    • Emergency Alerts: RP2040 can send alert codes for critical conditions
    • Status Query: RP2040 can query watchdog health to ensure external system is responsive
    • Protocol TBD: Communication method (UART, I2C, or CAN) to be defined in future

    9. Initialization Flow (setup())¶

    Startup sequence that initializes all subsystems.

    @startuml Initialization_Flow
start

:Initialize Serial @ 115200;
:Initialize Wire (I2C);
:Wire.setClock(400kHz);

:gnss.begin();
if (M9N responds?) then (no)
  :Print error, halt;
  stop
else (yes)
endif

:sensorMgr.addSensor(new Pt1000Sensor());
:sensorMgr.addSensor(new IMUSensor());
:sensorMgr.addSensor(new AltimeterSensor());

note right
  Sensors call begin()
  in addSensor()
end note

:watchdog.begin();

:Print boot banner;
:Wait for GNSS first fix;

:Enter main loop;
stop

@enduml

    Explanation:¶

    • Serial Init: Debug/status output at 115200 baud
    • I2C Init: Wire at 400 kHz on GPIO 4/5
    • GNSS Check: Must detect M9N at 0x42, otherwise halt
    • Sensor Registration: Sensors added to manager in order (calls their begin())
    • Watchdog Init: External system interface initialized
    • Boot Banner: User notification that system is alive
    • First Fix Wait: Wait for initial GNSS position lock before entering main loop

    10. Error Handling & Recovery¶

    State diagram showing system responses to various failure conditions.

    @startuml Error_Handling
state "Nominal" as nominal {
  state "Update GNSS"
  state "Update Sensors"
  state "Print Status"
}

state "Error Detection" as error_detect {
  state "I2C Timeout" as i2c_err
  state "Sensor Fault" as sensor_err
  state "No GNSS Fix" as gnss_err
  state "Watchdog Timeout" as watchdog_err
}

state "Recovery" as recovery {
  state "Retry I2C"
  state "Mark Sensor Invalid"
  state "Alert Watchdog"
  state "Log Error Code"
}

state "Safe Mode" as safe_mode {
  state "Continue with available sensors"
}

nominal --> error_detect: Detect anomaly

error_detect --> i2c_err
error_detect --> sensor_err
error_detect --> gnss_err
error_detect --> watchdog_err

i2c_err --> recovery: Attempt retry
sensor_err --> recovery: Disable sensor
gnss_err --> recovery: Continue (position optional)
watchdog_err --> recovery: Reset watchdog

recovery --> safe_mode: Mitigation complete
safe_mode --> nominal: Resume operation

note right of error_detect
  Errors detected via:
  - Timeout (I2C, GNSS read)
  - CRC failure (NMEA checksum)
  - Data out of range (pressure, temp)
  - Watchdog not responding
end note

note right of safe_mode
  System continues operating
  with degraded functionality
  Status shows "Invalid" for
  failed sensors
end note

@enduml

    Explanation:¶

    • Nominal Operation: Normal GNSS/sensor update and display cycle
    • Error Detection: I2C timeouts, sensor faults, GNSS loss, watchdog unresponsive
    • Recovery Mechanisms: Retry I2C, mark sensors invalid, alert external system
    • Safe Mode: Continue with available sensors rather than crashing
    • Logging: Errors recorded for post-flight analysis
